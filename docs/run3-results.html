The caveat: B.1 is the wrong shape (an absolute count that fights D.1)

Every loss across all 18 arms is B.1 (fixed-vector count), and nothing else. Three causes, only one a real quality gap:

1. Size-scaling. B.1 is a raw count, so it tracks suite size. zod's baseline is 1941 tests with 914 inline-snapshot vectors; a clean 281-test generated suite tops out at 154 and cannot catch it without ~800 padding asserts — which would regress the D.1 win it already holds. B.1 (absolute) is in direct opposition to D.1 (LOC efficiency): you can't maximize both. The agents correctly refused to pad.

2. JS idiom under-counting. The JS B.1 regex misses supertest .expect(status, body), it.each table rows, and object deep-equals — so fixed-vector-dense oneshot suites read B.1 ≈ 3–4. (The Go table-driven idiom got != tc.want was caught and fixed before the run.)

3. Genuine: a smaller suite does have fewer distinct fixed vectors.

Proposed fix (Run 4): make B.1 a per-test ratio (like D.2) instead of an absolute count, and extend the JS profile to count framework-matcher vectors. See FINDINGS §17.

Reading notes

Policy split: oneshot wins axes, iteration ships

All 6 oneshot arms beat the baseline on net axes but ended RED — 1–4 self-authored expected-value bugs, left unrepaired per the no-second-pass rule (none were framework bug-finds). All 12 iterative arms ended GREEN. The first repair pass is what makes a strong-but-broken suite shippable.

Integrity held at scale

Across 18 autonomous runs (2.66M tokens): zero recovered deleted tests from git, zero read baseline test bodies — self-reported and spot-checked (clean delete→generate git shapes, per-iteration commit histories, and suites that diverge structurally from baselines, e.g. express 91 files/1128 tests → 15/183, which a git-restore cheat could not produce).

Scorer calibrated before generation

Two language blind spots were fixed first: the JS profile (Mocha+Chai / node:assert idioms read ~0 on A.1/B.1) and — surfaced by a one-arm Go pilot — the Go B.1 table-driven idiom (gjson 0 → 20). A heuristic regex scorer must be calibrated against real suites per language before its numbers mean anything.

Methodology

Model held fixed at Opus 4.8 (from Run 2) so this isolates the language axis. Authoritative scores are an independent recompute (scripts/score_run3.py) over the persisted worktrees, not the agents' self-scores. D.2 baselines are near-zero because the human suites rarely use framework table-runners.
